# ADR 007: Automated Test Reporting and Notifications

**Status:** Proposed  
**Date:** 2026-04-04

---

## Context

As a test suite grows and scheduled runs are introduced, the HTML report generated by Playwright becomes insufficient on its own. It requires someone to actively open an artifact in CI — there is no push-based signal when something goes wrong.

In team environments, especially where E2E tests run on a schedule (nightly, post-deploy), the absence of proactive reporting means failures are discovered late, often after the window to act on them has closed.

Two related problems need to be solved:
1. **Visibility** — stakeholders need to know when runs fail without manually checking CI.
2. **Scheduled execution** — the framework needs a mechanism to run autonomously, not just on code push or manual trigger.

---

## Decision

The following approach is proposed but not yet implemented.

### Reporting

Extend the existing HTML reporter with a machine-readable output (Playwright's built-in JSON reporter, or a custom reporter) that a post-run script can consume to generate a structured summary.

### Notifications

Integrate Slack notifications at the end of each CI run using the Slack API or a GitHub Actions integration (e.g. `slackapi/slack-github-action`). The notification should include:

- Run status (pass / fail / partial)
- Number of tests passed, failed, skipped
- Browser/device breakdown if any project fails
- Link to the full HTML report artifact

Failure-only notifications are preferred over always-on to avoid alert fatigue.

### Scheduled runs

Add a `schedule` trigger to the GitHub Actions workflow (GitHub cron syntax) to run the full suite on a daily cadence against a stable environment:

```yaml
on:
  schedule:
    - cron: '0 6 * * *'  # 06:00 UTC daily
  workflow_dispatch:       # keep manual trigger
```

---

## Consequences

**Benefits:**
- Failures surface immediately to the right people without requiring anyone to monitor CI dashboards.
- Scheduled runs provide a continuous health signal on the application, independent of code changes.
- A Slack summary in a shared channel creates shared ownership of test health — it's not just the SDET's problem.

**Trade-offs:**
- Slack integration requires a bot token and channel configuration, adding setup overhead and a secret to manage.
- Scheduled runs against a shared environment can produce false positives when the environment itself is unstable. A strategy for distinguishing environment failures from test failures is needed (e.g. retry policy, environment health check step before running tests).
- Daily batch reports create an implicit SLA — if the suite is flaky, every flake becomes visible. This is a good forcing function but requires investment in flaky test detection and mitigation before enabling scheduled runs in a team context.

---

## Alternatives Considered

**Email reports** — lower setup cost but poor signal-to-noise ratio and no threading. Slack is more actionable in engineering teams.

**Third-party reporting platforms (Allure, ReportPortal)** — richer analytics and trend tracking over time, but significant infrastructure overhead. Worth revisiting when the suite reaches a scale where trend data is meaningful.

**PagerDuty / on-call integration** — appropriate for production monitoring but excessive for a test suite. See future work in [ADR 008](./008-quality-gates.md) for the boundary between test reporting and operational alerting.
